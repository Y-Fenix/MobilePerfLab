# MobilePerfLab

MobilePerfLab 是一款原创桌面移动端性能测试工具，工作流参考常见的移动性能分析产品：连接设备、选择 App、实时采集、曲线观察、事件标记、截图和报告导出。

## 当前能力

- Android 设备发现：通过 `adb devices -l` 自动识别真机。
- Android 应用选择：读取第三方包名列表，或自动识别前台应用。
- Android 实时采样：FPS/Jank、CPU、内存、电量、温度、估算功耗、网络速率；前台应用识别会组合 `dumpsys window`、`dumpsys activity` 等多路结果，CPU 会汇总同一包名的多个 PID，网络会优先读取 per-UID 统计，兼容 `uid_stat`、`xt_qtaguid` 和多种 `dumpsys netstats` 输出格式，失败时使用设备级上下行兜底并在日志中标注。
- Android 弱网工具：内置本机 HTTP/HTTPS 弱网代理，可一键设置/清除 Android 系统代理，支持延迟、抖动、丢包、上下行限速，并提供电梯、地铁、高速、隧道等场景预设。
- 弱网链路诊断：工具会读回 Android 当前系统代理，并从设备侧探测本机代理端口；若设备缺少 `nc`，会使用内置 HTTP 健康检查端点兜底，弱网页签会显示本机代理、Android 设备、设备代理、端口连通状态和弱网命中结论。
- 弱网安全清理：工具会记录自己写入过代理的 Android 设备，退出时自动尝试清理系统代理；弱网页签也可刷新当前代理状态。
- 低端机友好曲线：实时窗口默认开启展示层平滑，短暂缺采不会把曲线直接拉到 0；连续慢采样或采集异常增多时会自动切到“低端机保守展示”，让 FPS/CPU/功耗趋势更稳，同时明确标注展示策略。慢采样、兜底和受限样本会作为异常提示隔离，不会因为 adb 慢命令间隔变长而放大稳定展示曲线的波动。采样线程会按固定节拍尽量回到原有采样相位。报告中的曲线会同时展示“稳定展示”和“原始值”两条线，JSON 中同时保留 `samples` 原始采样和 `display_samples` 展示层数据。
- 采集健康状态：性能页会标出 FPS、CPU、内存、电量、温度、功耗、上下行网络的“正常 / 等待 / 无流量 / 异常”状态，并实时显示网络来源、异常样本占比和设备级网络兜底占比；“无流量”表示目标 App 当前没有业务请求，不等于网络采集失败。
- 网络可信标注：当 per-UID 和设备级网络计数都不可读时，工具会标注“网络采集不可用”；当目标 App UID 已识别且 per-UID 计数器可读但当前没有请求时，会按普通 0 流量/无流量处理，不会把整机网络误当目标 App 上下行。
- FPS 无新增帧标注：当 FPS 来源可用但采样窗口内没有新增帧时，实时界面和报告会标为“无新增帧”，用于区分静止页面/低端机短采样窗口和真正的 FPS 来源不可用。
- CPU 无增量标注：当 PID 可见但进程 CPU 计数没有变化时，实时界面和报告会标为“CPU 无增量”，用于区分进程空闲/短采样窗口和 PID 或 `/proc` 链路不可用。
- 采集链路自检：Android 可一键检查前台 App、PID、UID、FPS 来源和网络来源；开始采集前也会自动记录自检结果，导出的 JSON/HTML 会保留结构化自检表，用于定位“只有温度/内存”“FPS 取不到”“上下行取不到”等问题。
- 质量门禁：实时采集会按异常样本、设备级网络兜底、前台异常和慢采样给出“高可信 / 谨慎参考 / 不可信”的会话级判断，并额外显示“最近窗口”状态和“趋势来源”，用来判断当前这一小段曲线是稳定、节拍失稳、采集异常、网络兜底，还是更像真实性能波动。
- 实时质量事件：性能页底部会列出最新采集异常和设备级网络兜底事件，现场测试时无需等待报告即可定位不可信数据段。
- 曲线质量标识：实时曲线和 HTML 报告会标出异常样本与设备级网络兜底样本；橙色圆环表示网络数据来自设备级兜底，红色三角表示该采样点存在采集异常说明，浅橙/浅红背景表示连续兜底或异常区间。
- iOS 设备发现：优先使用 `pymobiledevice3 usbmux` 识别真实在线设备，再用 Xcode `devicectl`/`xctrace` 补充状态。
- iOS 基础采样：通过 `pymobiledevice3 diagnostics battery` 读取电量、温度、估算功耗。
- iOS 进程采样：启动 tunnel 后通过 `pymobiledevice3 developer dvt sysmon` 读取 CPU/内存。
- iOS FPS 采样：启动 tunnel 后通过 `pymobiledevice3 developer dvt graphics` 读取 `CoreAnimationFramesPerSecond`，设备必须能被 `pymobiledevice3 usbmux list` 识别。
- iOS Jank：当前按 FPS 相对 60 FPS 的掉帧比例估算，用于趋势参考。
- 演示模式：没有真机时可预览完整 UI、实时曲线、标记和导出流程。
- 报告导出：一次导出 CSV、JSON、HTML 三份结果；HTML 报告包含质量门禁、指标可用性、实机验证清单、修复建议、采样节拍、采集链路自检、异常/兜底样本占比、异常区间摘要、弱网链路诊断和网络数据来源，设备级网络兜底会明确标注“非目标 App 独占流量”。

## 运行

双击 `一键启动.command`，或在终端执行：

```bash
python3 mobileperflab.py
```

## 测试

```bash
python3 -m unittest discover -s tests
python3 -m py_compile mobileperflab.py
```

`tests/` 目录建议一起上传到 GitHub，用于回归验证曲线稳定、采集健康状态、Android 前台应用识别、多进程 CPU 采样、上下行网络解析、弱网代理诊断和报告质量摘要。

## 打包

双击 `一键打包.command`。打包完成后生成：

```text
dist/MobilePerfLab.app
```

## Android 环境

推荐安装 Android SDK Platform-Tools，并确保 `adb` 在 `PATH` 中。也可以把 `platform-tools/adb` 放在项目目录或同级 `AndroidTools/platform-tools/adb` 下，应用会自动尝试识别。

设备需开启 USB 调试并授权当前电脑。

启动后侧栏会显示环境自检结果，日志里会列出 Python、Android ADB、iOS `pymobiledevice3`、Xcode `xcrun` 的可用状态。缺少 `adb` 时仍可使用演示模式和 iOS 基础能力，但 Android 真机采集不可用。

### Android 采集自检

选择 Android 设备并填写目标包名后，可以点击“采集自检”。工具会检查：

- 前台：当前前台 App 是否等于目标包名。
- PID：是否能通过 `pidof` / `pgrep` / `ps` 找到目标进程。
- UID：是否能通过 `dumpsys package`、`/proc/<pid>/status` 或 `pm list packages -U` 找到 App UID。
- FPS：当前可用的帧率来源，例如 `gfxinfo counters`、`gfxinfo framestats` 或 `SurfaceFlinger`。
- 网络：是否能使用目标 App per-UID 上下行；工具会依次尝试 `uid_stat`、`xt_qtaguid` 和 `dumpsys netstats`。如果 per-UID 计数器可读但当前为 0，会显示为目标 App 暂无流量；只有 per-UID 不可读时才会使用设备级网络兜底，界面和报告都会标注“非目标 App 独占流量”。

如果只看到温度、内存等少数指标，先看“采集链路”和“采集健康”两行状态。常见原因是目标 App 不在前台、目标 PID/UID 无法识别、系统限制 `/proc` 或 `netstats`、页面静止导致 FPS 没有新增帧。

### 数据可信度

实时界面和报告使用同一套质量规则：

- 高可信：采集链路稳定，异常样本和兜底样本较少。
- 谨慎参考：存在一定比例的异常样本、设备级网络兜底或采样节拍波动。
- 不可信：前台异常、慢采样过多、异常样本过多或可信度过低。

实时曲线默认会做展示层平滑，目的是降低低端机短暂缺采造成的视觉抖动；CSV/JSON/HTML 仍保存原始采样点。HTML 报告中粗线为“稳定展示”，细线为“原始值”；JSON 中 `samples` 为原始值，`display_samples` 为展示层平滑值。当实时质量发现慢采样或采集异常连续出现时，工具会从“标准稳定展示”自动切到“低端机保守展示”，进一步限制 FPS、CPU 和功耗的单点跳变，报告中的 `quality.display_strategy` 会记录本次使用的展示策略。实时质量栏会根据最近窗口直接提示“采样间隔调到 1.5s / 2.0s，优先看稳定展示”“按真实性能波动分析”等下一步动作；建议值会按当前采样间隔逐步给出。采样线程按固定节拍调度，单次 adb 慢命令结束后会尽量追回原采样相位，避免后续曲线持续漂移；如果低端机上把采样间隔调到 1.5s 或 2s，实时质量和报告会按新的间隔判断慢采样，不会把主动降压误判成节拍失稳。Android FPS 计数器和上下行网络这类 delta 指标会按实际计数器读取时刻计算速率，减少低端机 adb 慢命令导致的 FPS/网络速率误放大。报告中的“采样节拍”会根据样本间隔判断“节拍稳定 / 节拍波动 / 节拍失稳”，用于区分真实性能波动和采集慢命令造成的曲线波动。

切后台再切回目标 App 时，Android 的 Surface、进程 CPU 和网络 delta 基线需要重新建立。实时界面会把这段标为“恢复中 / 前台恢复窗口”，而不是直接判定 FPS、CPU 或网络不可用；这段样本会在曲线和报告中单独标注，建议不要用于性能结论。

### 报告可信度

HTML 报告会生成“实机验证清单”，按链路给出通过、注意、失败或待验证：

- FPS 链路：判断 FPS 是否存在未采集或无帧增量。
- CPU 链路：判断 CPU 是否存在无进程增量或采集失败。
- 网络链路：区分目标 App per-UID、设备级兜底和不可用。
- 前后台一致性：标记目标 App 离开前台或刚切回前台的恢复窗口。
- 采样节拍：标记低端机或 adb 慢命令造成的慢采样。
- 弱网命中：导出时结合弱网代理是否捕获到真实请求判断。

Android 报告还会包含“采集链路自检”区块，记录前台应用、PID/UID 来源、FPS 来源和网络来源。这个区块用于解释报告曲线为什么可信或为什么需要谨慎参考。

HTML/JSON 报告还会生成“指标可用性”，按 FPS、CPU、内存、温度、功耗、下行和上行展示有效样本覆盖率、来源和状态。低端机只采到内存/温度时，这里会直接标出其它指标不可用、兜底或待验证，避免误读曲线。

HTML/JSON 报告还会生成“会话可用性”结论：当 FPS、CPU 或上下行网络不可用时，会明确标记“只可参考部分指标”，提示这份报告不能用于判断流畅度、CPU 占用或目标 App 网络。

HTML/JSON 报告还会生成“最近窗口”，默认看最后 8 个样本，统计慢采样、采集异常、设备级网络兜底数量和 FPS 波动幅度。它会给出“趋势：性能波动 / 采集波动 / 平稳”等结论，并同步输出“采样建议”，用于解释报告末段曲线是否还能继续参考；全局质量门禁看整场，最近窗口看当前这段。

HTML/JSON 报告还会生成“修复建议”，把失败、注意和待验证项转换成下一步动作，例如保持目标页面可见并检查 `gfxinfo`/`SurfaceFlinger`、确认 per-UID 网络来源、低端机调大采样间隔，或确认弱网是否命中系统 HTTP/HTTPS 代理。

## 弱网工具

主工作区包含“性能采集”和“弱网工具”两个页签。“弱网工具”提供常用网络预设，也可以手动配置端口、延迟、抖动、丢包率、下行和上行限速。

Android 使用方式：

1. 选择 Android 设备。
2. 切到“弱网工具”页签，选择预设或填写参数。
3. 点击“启动代理”。
4. 点击“应用到 Android”，工具会通过 adb 写入系统 HTTP 代理。
5. 查看“链路诊断”，确认“设备代理”为“已确认”，“端口连通”为“可达”。
6. 测试结束后点击“清除代理”，避免设备继续走代理。

弱网页的“代理真实流量”会实时显示经过本机弱网代理的上下行速率、累计流量、连接数和丢弃数，并结合链路诊断给出“弱网已生效 / 等待目标流量 / 疑似绕过代理 / 代理未确认 / 端口不可达”等命中结论。性能页顶部会同步显示短行动提示，例如“弱网：可以开始测试”“弱网：先触发业务请求”“弱网：先修弱网链路”，便于边看性能曲线边判断弱网是否已经可用。它用于确认目标场景是否真的走到了弱网代理；该数据是代理级统计，不等同于目标 App 独占网络流量。导出报告时也会记录弱网代理运行配置、真实流量快照、历史曲线、Android 代理读回状态、端口连通状态、命中评分和测试就绪判断。

测试就绪判断会把弱网状态归纳成三种下一步动作：“可以开始测试”表示弱网链路和真实流量已经确认；“先触发业务请求”表示代理链路已就绪但还没看到目标请求；“先修弱网链路”表示代理未启动、端口不可达、设备代理未确认或疑似绕过代理。报告还会对比目标 App 上下行峰值和弱网代理峰值，如果 App 流量远高于代理流量，会标记“目标流量待确认”，避免把无关代理流量误判为弱网已命中目标 App。

导出报告时还会对比性能采集中的 App 上下行和弱网代理真实流量：如果 App 已经有网络速率，但代理仍是“等待目标流量”，报告会标记“疑似绕过系统代理”，并提示检查 QUIC/UDP、自建网络栈、代理白名单或证书/代理配置。

链路诊断含义：

- 本机代理：桌面端弱网代理是否正在监听。
- Android 设备：当前是否选择了 Android 设备。
- 设备代理：adb 写入后再读回的 Android 系统代理是否等于当前代理地址。
- 端口连通：Android 设备是否能连到电脑上的弱网代理端口；工具会先尝试 `toybox nc` / `nc`，再尝试访问 `http://<host>:<port>/__mobileperflab_health`。

当前弱网为本机 HTTP/HTTPS 代理模式，不需要 Root，适合大部分走系统代理的接口请求。若目标 App 使用 UDP、QUIC，或主动绕过系统代理，则需要后续接入 VPN/tun 模式才能完全覆盖。

报告中的弱网状态含义：

- 已命中目标流量：弱网代理捕获到真实请求。
- 等待目标流量：代理运行中，但导出时没有捕获到目标请求，需要确认 App 是否走系统代理。
- 已命中并丢弃：弱网丢包规则触发，建议结合业务日志确认是否命中目标请求。

如果忘记清除代理，工具退出时会自动尝试清理本次由工具写入过代理的 Android 设备。建议测试结束仍手动点击“清除 Android 代理”，并用“刷新状态”确认设备当前代理状态。

## iOS 环境

iOS 真机需要先信任电脑。基础识别可使用 Xcode 命令行工具；更完整的设备信息建议安装：

双击 `安装iOS依赖.command` 安装 `pymobiledevice3`。

如果要采集 iOS CPU/内存/FPS，双击 `启动iOS采集服务.command`，输入本机登录密码，并保持该窗口打开。FPS 来自 DVT graphics 的 `CoreAnimationFramesPerSecond`；如果日志提示 `pymobiledevice3 当前未识别设备`，请重新插拔、信任设备，或切换到 `pymobiledevice3 usbmux list` 可见的设备。
