# MobilePerfLab 对标专业性能工具差距与 UI 优先改造 PRD

## 1. 背景

MobilePerfLab 当前已经具备 Android/iOS 性能采集、低端机稳定展示、报告导出和 HTTP/HTTPS 弱网代理能力，但整体体验仍偏工程工具：入口多、状态文案重、视觉层级弱、图表解读成本高。用户的最新目标是先把 UI 和使用体验往 PerfDog 一类专业移动性能工具靠近，再继续补齐弱网工具和零侵入体验。

本项目只参考公开移动性能测试工具的通用工作流和信息架构，不使用 PerfDog 品牌、图标、界面素材、私有协议或未授权资源。

## 2. 本轮目标

1. 先完成差距梳理和需求文档，再进入开发。
2. UI 优先：把主界面改成专业测试工作台，0 学习成本完成连接设备、选择应用、开始采集、看曲线、定位异常、导出报告。
3. 尽量做到 0 入侵：Android 不 Root，iOS 不越狱；需要脚本或后台服务时由工具静默启动、静默检查、失败时给出明确修复动作。
4. 弱网工具作为一等功能，而不是隐藏附属面板；需要能配置、应用、诊断、证明命中。
5. 每个功能必须有自测：优先自动化测试，其次可重复手动验收步骤。
6. 开发期间默认不主动打开打包应用，避免影响用户桌面；只有用户明确要求运行查看时才打开。
7. 每个阶段汇报：完成内容、自测证据、累计 token、累计耗时。

## 3. 公开对标能力摘要

从公开产品资料和常见移动性能工具工作流抽象出以下能力，不复刻任何私有实现：

- 连线即用：连接手机后识别设备和 App，尽量少让用户手动执行命令。
- 关键指标集中展示：FPS、Jank、CPU、内存、网络、温度、功耗等实时趋势清晰可读。
- 专业任务流：设备选择、应用选择、开始/停止、打点、报告导出应在首屏可见。
- 数据可信度表达：采集不可用、前后台切换、兜底来源、恢复窗口要明确标记。
- 弱网验证闭环：能配置网络条件，应用到设备，并证明目标流量是否经过弱网链路。

## 4. 当前差距

### 4.1 UI 视觉层级

现状：

- Tk 默认控件痕迹明显，边框、按钮、卡片和表格风格偏原型工具。
- 顶部、侧栏、指标卡、图表、日志区域的信息密度接近，但主次不够强。
- 状态文案偏长，容易让用户在首屏“读说明”，而不是直接操作。

差距：

- 专业工具应该像测试控制台：当前会话状态、设备/App、开始按钮、关键指标和异常状态一眼可见。
- 常用动作需要更像工具栏和状态栏，不应散落在多个面板中。

### 4.2 首屏任务流

现状：

- 左侧能选择设备、App、采样间隔，也有采集自检和弱网简版参数。
- 性能采集和弱网工具通过页签区分，但弱网状态同步展示仍不够突出。

差距：

- 用户需要明确看到“1 连接设备 -> 2 选择 App -> 3 开始采集 -> 4 观察/打点 -> 5 导出”的自然路径。
- 环境缺失、设备未授权、App 未选择、DVT/tunnel 缺失等状态应在路径上就地提示，不应主要依赖日志。

### 4.3 数据可视化

现状：

- 已有 8 个指标图表，默认可见 4 个，滚轮按行滚动。
- 有原始值、稳定展示、质量标记、报告双曲线。

差距：

- 图表阅读仍偏“看线”，缺少专业工具常见的当前值、均值、峰值、异常区间摘要和图例固定表达。
- 图表之间缺少统一时间轴和明显的会话标记感。
- 质量标记在图上存在，但用户还需要更轻量地理解“这是采集问题、设备兜底、恢复窗口还是真实波动”。

### 4.4 0 入侵和静默依赖

现状：

- Android 基于 adb，不 Root。
- iOS 基础能力可用，DVT/tunnel 需要额外脚本。
- README 说明需要用户双击 iOS 采集服务脚本。

差距：

- 用户预期是连线后尽量自动处理依赖；如果必须运行服务，应由应用静默拉起，并在 UI 内展示状态。
- 失败时需要给“一键重试/打开修复动作”，不要让用户去 README 里找命令。

### 4.5 弱网工具

现状：

- 已有本机 HTTP/HTTPS 代理、预设、延迟/抖动/丢包/上下行限速、Android 代理写入/清除、端口连通和真实代理流量。

差距：

- 弱网页 UI 仍像配置表单，不像专业弱网测试台。
- 需要更明确的“链路状态灯”：代理监听、设备代理、端口连通、真实流量、目标 App 命中。
- iOS 只能提示手动配置，后续需要更清楚地展示配置二维码/步骤或系统限制。

### 4.6 质量和进度透明

现状：

- 自动化测试覆盖较多；当前 258 个测试通过。
- 报告包含质量门禁和验证清单。

差距：

- 开发过程缺少阶段性进度文档。
- 每次功能完成后需要固定汇报“变更、验证、token、耗时、剩余风险”。

## 5. UI 优先改造需求

### 5.1 信息架构

主界面改为四层结构：

1. 顶部会话栏：设备状态、目标 App、采集状态、弱网状态、报告入口。
2. 左侧控制栏：设备、App、采样、环境/依赖、主操作按钮。
3. 中央观测区：指标总览、核心曲线、标记和时间轴。
4. 右侧诊断栏：采集链路、质量事件、弱网命中、日志摘要。

验收标准：

- 未开始采集时，用户能在首屏按顺序完成设备选择、App 选择和开始采集。
- 采集中，用户不切页也能看到采集状态、弱网状态和数据可信度。
- 不依赖日志即可判断当前缺什么。

### 5.2 视觉规范

要求：

- 专业测试工具风格：克制、清晰、高密度，不做营销页。
- 颜色用于状态和指标，不做大面积装饰。
- 卡片圆角不超过 8px，避免嵌套卡片。
- 按钮区分主动作、次动作和危险动作。
- 状态使用短标签：可用、等待、受限、兜底、异常、恢复中。

验收标准：

- 首屏不出现大段说明文字。
- 关键按钮和状态不被图表或日志挤压。
- 所有按钮文字在常见窗口宽度下不截断。

### 5.3 指标卡

每个指标卡显示：

- 当前展示值。
- 单位。
- 状态短标签。
- 来源/异常一句话。
- 小型趋势或变化提示：最近窗口均值/峰值/是否异常。

指标包括：

- FPS
- Jank
- CPU
- Memory
- Temperature
- Power
- Down
- Up

验收标准：

- 指标不可用时显示“不可用/原因”，不能显示像真实 0 的误导值。
- per-UID 0 流量显示“无流量”，不是“异常”。
- FPS 无新增帧显示“无新增帧”，不是“FPS 不可用”。

### 5.4 图表区

要求：

- 默认展示 2 行 x 2 列，即 4 个图表。
- 支持滚轮一次滚动 1 行。
- 每个图表显示当前值、单位、采样状态、峰值/均值摘要。
- 统一时间窗和标记线。
- 异常、兜底、受限、恢复窗口使用固定图例。

验收标准：

- FPS、CPU、内存、上下行网络优先展示；温度、功耗、Jank 可继续滚动查看。
- 图表数据使用原始点保存，展示层稳定曲线只用于可视化。
- 最近窗口恢复稳定后，保守展示能恢复到标准稳定展示。

### 5.5 采集链路诊断

要求：

- 右侧固定展示前台、PID、UID、FPS、网络来源状态。
- “采集自检”可手动触发，开始采集前自动触发。
- 链路异常时提供明确动作。

验收标准：

- Android 只采到温度/内存时，UI 明确指出 FPS/CPU/网络各自失败原因。
- 切后台/切回来时显示“目标不在前台”和“前台恢复窗口”，恢复窗口不参与性能结论。

### 5.6 弱网工具 UI

弱网作为独立工作台，但状态同步到性能页顶部。

弱网页结构：

1. 场景预设：正常、弱网、极弱网、电梯、地铁、高速、隧道、自定义。
2. 参数区：延迟、抖动、丢包、下行、上行、端口。
3. 应用区：启动代理、应用到 Android、清除代理、刷新状态。
4. 状态灯：代理监听、设备代理、端口连通、代理流量、目标命中。
5. 流量图：代理下行、代理上行、丢弃数、连接数。
6. 就绪结论：可以开始测试、先触发业务请求、先修弱网链路、疑似绕过代理。

验收标准：

- 用户能在 3 步内完成 Android 弱网：选择预设 -> 启动并应用 -> 看就绪状态。
- 有 App 流量但代理无流量时，明确提示疑似绕过代理。
- 代理有流量但 App 上下行未确认时，提示目标待确认，不能宣称命中目标 App。

### 5.7 0 入侵和静默运行

要求：

- Android：继续 adb 模式，不 Root；自动发现 adb。
- iOS：不越狱；需要 pymobiledevice3/tunneld/DVT 时由应用检测并尝试静默启动。
- 静默启动失败时只在 UI 给出操作按钮和错误原因，不弹出长期挂着的额外窗口。

验收标准：

- 开发和测试阶段不主动打开打包应用，除非用户明确要求。
- iOS 服务未就绪时，UI 显示“可尝试自动启动/手动修复”，不要求用户先读 README。

## 6. 开发优先级

### P0：UI 工作台重排

- 顶部会话栏。
- 左侧步骤化控制栏。
- 中央指标卡 + 图表区域。
- 右侧诊断栏。
- 不改变采集核心逻辑，先把现有能力重新组织。

### P1：图表可读性增强

- 图表摘要：当前、均值、峰值。
- 固定图例。
- 统一状态标记。
- 最近窗口结论更贴近图表。

### P2：弱网工作台增强

- 弱网状态灯。
- 3 步操作路径。
- 代理流量图和目标命中结论更突出。

### P3：静默依赖和 0 入侵完善

- iOS DVT/tunnel 静默检测和尝试启动。
- 失败修复动作 UI 化。

## 7. 自测要求

每个功能至少包含：

- 自动化测试：纯函数、状态计算、布局契约或报告 payload。
- 静态验证：`python3 -m py_compile mobileperflab.py`。
- 全量回归：`python3 -m unittest discover -s tests`。
- UI 改造如无法自动截图，必须提供不打开桌面应用的布局契约测试；需要打开应用时必须先征得用户明确同意。

当前固定验证命令：

```bash
python3 -m unittest discover -s tests
python3 -m py_compile mobileperflab.py
git diff --check
```

## 8. 开发进度汇报格式

每完成一个功能或阶段，必须汇报：

- 阶段名称。
- 完成内容。
- 自测证据。
- 提交 hash。
- 当前累计 token。
- 当前累计耗时。
- 剩余风险/下一步。

## 9. 非目标

- 不复制 PerfDog 商标、图标、配色、私有协议、云平台或账号体系。
- 不默认 Root/Jailbreak。
- 不在用户未同意时打开打包应用做长时间视觉检查。
- 不先做大规模架构重写；UI 工作台优先使用当前 Tk 代码可落地的重排和组件化。

## 10. 第一阶段验收清单

P0 完成后应满足：

- 首屏形成专业测试工作台，而不是配置表单堆叠。
- 用户能 0 学习成本完成“连接设备 -> 选择 App -> 开始采集 -> 看核心曲线 -> 导出报告”。
- 采集质量、弱网状态和链路诊断在采集中始终可见。
- 不可用指标不会伪装成真实 0。
- 全量测试通过，并新增 UI 布局/状态契约测试。
