# MobilePerfLab 性能采集与弱网工具 PRD

## 1. 背景

MobilePerfLab 是桌面端 Android/iOS 性能测试与弱网测试工具。产品体验参考 PerfDog 一类移动性能分析工具的通用工作流：连接设备、选择或识别前台应用、开始采集、实时观察曲线、标记异常、导出报告、结合弱网验证。

本项目不得使用 PerfDog 的品牌、图标、界面素材、私有协议或未授权资源。目标是复刻移动性能测试工具的通用信息架构、使用节奏和数据可信度表达。

## 2. 目标

1. 提供可直接运行的桌面工具，覆盖性能采集和弱网测试两类核心任务。
2. Android 与 iOS 均能自适应展示设备、应用、指标可用性和异常原因。
3. 低端 Android 设备采样不稳定时，曲线展示尽量接近中高端设备的可读性，同时保留原始采样，不伪造数据。
4. 弱网工具可配置延迟、抖动、丢包、上下行限速，并能证明弱网是否真实命中目标链路。
5. 导出报告能解释数据可信度，避免把采集异常、设备级兜底或弱网绕过误判为目标应用性能问题。
6. 每个功能上线前必须有对应自测证据，优先自动化测试，其次提供可重复的手动验收步骤。

## 3. 非目标

1. 不实现 PerfDog 账号、云平台、团队协作或商业化能力。
2. 不使用 Root、Jailbreak 或私有移动系统接口作为默认依赖。
3. 一期弱网不覆盖 UDP、QUIC、自建网络栈和 VPN/tun 全局代理；这些链路需要在报告中明确提示可能绕过系统代理。
4. iOS 上不承诺所有指标在无 DVT/tunnel 环境下都可采集；需要显示限制和修复动作。

## 4. 用户与场景

### 4.1 测试人员

需要在真机上快速完成性能回归、弱网回归和报告交付。重点关注是否能快速开始、能否看懂曲线、报告是否能说明可信度。

### 4.2 开发人员

需要定位 FPS、CPU、内存、网络、温度、功耗异常。重点关注原始数据、采集来源、异常区间和修复建议。

### 4.3 低端机专项测试

需要处理 adb 慢命令、帧计数短窗口无增量、前后台切换、网络 per-UID 不可读等情况。重点关注曲线不要被采集噪声拉爆，同时不能隐藏真实波动。

## 5. 核心工作流

1. 启动应用后默认全屏打开主窗口。
2. 左侧完成环境自检、设备选择、应用选择、采样间隔选择和开始/停止采集。
3. 用户可点击“当前前台应用”自动填入包名，也可手动输入。
4. 开始采集前自动执行 Android 采集链路自检，记录前台、PID、UID、FPS 来源和网络来源。
5. 采集中实时展示指标卡、趋势曲线、采集质量、弱网状态和质量事件。
6. 用户可切到弱网工具页签，选择预设或手动参数，启动代理并应用到 Android。
7. 弱网运行期间实时展示代理上下行、连接数、丢弃数、链路诊断和命中结论。
8. 停止采集后导出 CSV、JSON、HTML 报告。
9. 测试结束时可一键清除 Android 代理；应用退出时也要尝试清理本次写入过的代理。

## 6. 性能采集需求

### 6.1 设备识别

Android：

- 使用 adb 识别在线设备、型号、系统版本和授权状态。
- 未授权、离线或 adb 不存在时，需要在 UI 明确提示。

iOS：

- 优先使用 pymobiledevice3 识别在线设备。
- 可用时结合 Xcode 工具补充状态。
- DVT/tunnel 不可用时，需要说明 CPU、内存、FPS 等指标受限。

### 6.2 应用识别

- Android 支持读取包名列表。
- Android 支持识别当前前台 App。
- 采集中需要轻量探测前台应用，快速识别切后台或切回目标 App。
- 目标 App 不在前台时，当前样本必须标注“目标应用不在前台”。
- 目标 App 切回前台后，FPS、CPU、网络 delta 基线需要重建，并标注“前台恢复窗口”。

### 6.3 指标

Android 必须尽力采集：

- FPS
- Jank
- CPU
- Memory
- Battery
- Temperature
- Power
- Network Down
- Network Up

iOS 一期支持：

- Battery
- Temperature
- Power
- DVT 可用时采集 CPU、Memory、FPS
- Jank 可按 FPS 与目标帧率的差异估算并标明来源

### 6.4 数据来源与可信度

- FPS 可来自 gfxinfo counters、gfxinfo framestats 或 SurfaceFlinger。
- CPU 需要支持同一包名多个 PID 汇总。
- 网络优先使用目标 App per-UID 统计。
- per-UID 不可用时允许使用设备级网络兜底，但 UI 和报告必须标注“非目标 App 独占流量”。
- per-UID 可读但当前为 0 时，应显示“无流量”，不能误判为采集失败。
- FPS 来源可用但没有新增帧时，应显示“无新增帧”，不能误判为 FPS 来源不可用。
- PID 可见但 CPU 计数无变化时，应显示“CPU 无增量”，不能误判为 PID 丢失。

## 7. 低端机曲线需求

### 7.1 原则

- 原始采样必须保留，不能被展示层平滑覆盖。
- 实时曲线默认展示稳定曲线，让短暂缺采、慢采样和恢复窗口更易读。
- 报告必须同时保留原始曲线和稳定展示曲线。
- 稳定展示只用于可视化，不作为原始数据。

### 7.2 低端机保守展示

当满足任一条件时，自动进入低端机保守展示：

- 连续慢采样。
- 最近窗口异常样本增多。
- FPS/CPU/网络 delta 指标处于恢复窗口。
- 采样耗时超过当前采样间隔并影响曲线节拍。

保守展示需要：

- 限制 FPS、CPU、功耗的单点跳变。
- 慢采样不应因为时间间隔变长而放大速率类指标。
- 前台恢复窗口样本不参与性能结论。
- 质量栏明确显示当前展示模式。

### 7.3 采样间隔建议

- 默认采样间隔为 1.0s。
- 低端机或慢 adb 场景下，实时质量栏可建议调到 1.5s 或 2.0s。
- 用户主动调大采样间隔后，质量规则需要按新的间隔判断慢采样。

## 8. 弱网工具需求

### 8.1 弱网模式

一期使用本机 HTTP/HTTPS 代理模式。

参数：

- 监听端口
- 延迟
- 抖动
- 丢包率
- 下行限速
- 上行限速

预设：

- 正常网络
- 弱网
- 极弱网
- 电梯
- 地铁
- 高速
- 隧道

### 8.2 Android 自动配置

- 支持一键写入 Android 系统 HTTP 代理。
- 支持读回当前 Android 代理状态。
- 支持从设备侧探测本机代理端口连通性。
- 支持一键清除 Android 代理。
- 应用退出时自动尝试清理本次写入过代理的设备。

### 8.3 iOS 说明

- 一期不自动写入 iOS Wi-Fi 代理。
- UI 需要提示用户手动在 Wi-Fi 设置中配置 HTTP 代理。

### 8.4 弱网命中判断

弱网状态需要区分：

- 弱网未启动
- 代理未确认
- 端口不可达
- 等待目标流量
- 已命中目标流量
- 已命中并发生丢弃
- 疑似绕过代理
- 目标流量待确认

判断依据：

- 代理是否监听。
- Android 系统代理是否等于当前代理。
- 设备是否能访问代理端口。
- 代理是否捕获真实流量。
- 代理上下行与目标 App 上下行是否同步。
- 是否出现 App 有流量但代理无流量。

## 9. UI 与交互需求

### 9.1 主窗口

- 启动后全屏打开。
- 视觉风格应偏专业测试工具，避免营销页式布局。
- 信息密度高但分区清晰。
- 不使用 PerfDog 品牌素材。

### 9.2 主布局

- 左侧：设备、应用、采样控制、采集自检、环境提示。
- 顶部：会话状态、采集质量、弱网状态、报告入口。
- 中部：指标卡和实时曲线。
- 右侧或底部：质量事件、弱网摘要、日志。

### 9.3 图表

- FPS、Jank、CPU、Memory、Temperature、Power、Network Down、Network Up 都需要图表。
- 右侧图形区每行展示 2 个图表，默认可见 4 个图表。
- 超出可见区域后通过鼠标滚轮滚动。
- 滚轮每次滚动显示上一行或下一行，不按固定像素滚动。
- 图表必须显示质量标记：异常、兜底、恢复窗口。
- 图表不可被右侧/底部面板遮挡。

### 9.4 指标卡

每个指标卡需要显示：

- 当前展示值
- 单位
- 健康状态
- 来源或异常原因

### 9.5 弱网页签

需要包含：

- 预设选择
- 参数输入
- 启动/停止代理
- 应用到 Android
- 清除 Android 代理
- 链路诊断
- 真实代理流量曲线
- 测试就绪判断

## 10. 报告需求

导出格式：

- CSV 原始采样
- JSON 结构化数据
- HTML 可视化报告

HTML 报告需要包含：

- 会话摘要
- 质量门禁
- 指标可用性
- 最近窗口趋势
- 采样节拍
- 采集链路自检
- 原始曲线与稳定展示曲线
- 异常区间
- 质量事件
- 弱网配置
- 弱网链路诊断
- 弱网真实流量
- 弱网命中结论
- 修复建议

报告结论需要明确区分：

- 性能波动
- 采集波动
- 网络来源不可信
- 弱网未命中
- 只可参考部分指标

## 11. 自测要求

### 11.1 自动化测试

每次提交前必须执行：

```bash
python3 -m unittest discover -s tests
python3 -m py_compile mobileperflab.py
git diff --check
```

新增或修改功能必须优先补测试：

- Android 前台识别
- FPS/CPU/网络解析
- 低端机曲线稳定展示
- 采集质量门禁
- 弱网诊断
- 弱网报告
- HTML 报告关键字段
- UI 纯逻辑函数

### 11.2 手动验收

每个可视化或设备相关功能需要提供手动验收步骤。至少覆盖：

- 无设备时可进入演示模式。
- Android 真机可识别设备和前台 App。
- 目标 App 切后台再切回，曲线显示恢复窗口。
- 低端机慢采样时出现保守展示与采样间隔建议。
- 弱网代理可启动、应用到 Android、读回代理、检测端口。
- 触发目标 App HTTP/HTTPS 请求后，弱网真实流量曲线变化。
- 导出报告后，HTML 中能看到质量门禁、指标可用性、弱网结论。

## 12. 验收标准

1. 用户双击一键启动脚本即可打开主窗口。
2. 主窗口默认全屏。
3. 无真机时演示模式可完整预览性能页、弱网页和报告导出。
4. Android 真机上至少能明确显示每个指标是“正常、等待、无流量、无新增帧、无增量、兜底或异常”。
5. 当设备只能采到温度和内存时，界面和报告能指出 FPS、CPU、网络不可用原因。
6. 前后台切换后，FPS、CPU、网络不会沿用旧 delta 基线误算。
7. 弱网工具能显示代理是否真实命中目标流量。
8. 报告曲线比单条原始折线更直观，并保留原始数据。
9. 全量自动化测试通过。
10. 每个开发批次都记录已执行的自测命令和结果。

## 14. 当前已知风险

1. Android 厂商 ROM 可能限制 `/proc`、`netstats`、`SurfaceFlinger` 或 `gfxinfo`。
2. 目标 App 使用 UDP、QUIC 或自建网络栈时，HTTP/HTTPS 系统代理可能无法命中。
3. iOS DVT 依赖 pymobiledevice3、tunneld 和系统权限，用户环境差异大。
4. 低端机性能真实波动与采集慢命令噪声可能同时存在，需要依赖质量门禁和原始数据共同判断。
5. 单文件应用已较大，后续改动需要控制范围，并逐步抽离纯逻辑以提升可测性。
